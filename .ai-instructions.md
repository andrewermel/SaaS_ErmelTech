## 1. VISÃO GERAL DO PROJETO

### O que é

Aplicação full-stack para gerenciamento de lanchonetes. Atualmente é single-tenant (um único dono, sem isolamento de dados). O objetivo é transformá-la em um **SaaS multi-tenant**, onde cada empresa (lanchonete) terá seus dados isolados.

### Origem

O projeto é uma cópia do repositório original "BlackLanches". Todas as referências a "blacklanches" foram migradas para "ermeltech".

**Todos os nomes (containers Docker, credenciais do banco, network, package names e frontend) já foram atualizados para `ermeltech`/`saas_ermeltech-*`.**

### Stack

| Camada   | Tecnologia                        |
| -------- | --------------------------------- |
| Backend  | Node.js 20, Express 5, TypeScript |
| ORM      | Prisma (PostgreSQL)               |
| Auth     | JWT (jsonwebtoken) + bcryptjs     |
| Frontend | React 18, Vite, CSS customizado   |
| Banco    | PostgreSQL 16                     |
| Testes   | Jest + Supertest                  |
| Infra    | Docker, Docker Compose            |
| CI/CD    | GitHub Actions                    |

## 3. BANCO DE DADOS ATUAL (schema.prisma)

```prisma
model User {
  id            Int           @id @default(autoincrement())
  name          String
  email         String        @unique
  password      String
  createdAt     DateTime      @default(now())
  userCompanies UserCompany[]
}

model Company {
  id          Int           @id @default(autoincrement())
  name        String
  slug        String        @unique
  email       String?
  phone       String?
  users       UserCompany[]
  ingredients Ingredient[]
  portions    Portion[]
  snacks      Snack[]
  createdAt   DateTime      @default(now())
  updatedAt   DateTime      @updatedAt
}

enum Role {
  OWNER
  ADMIN
  EMPLOYEE
}

model UserCompany {
  id        Int      @id @default(autoincrement())
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  userId    Int
  company   Company  @relation(fields: [companyId], references: [id], onDelete: Cascade)
  companyId Int
  role      Role     @default(EMPLOYEE)
  createdAt DateTime @default(now())
  @@unique([userId, companyId])
  @@index([userId])
  @@index([companyId])
}

model Ingredient {
  id        Int       @id @default(autoincrement())
  company   Company?  @relation(fields: [companyId], references: [id], onDelete: Cascade)
  companyId Int?      // ⚠️ Nullable para compatibilidade com dados pré-SaaS
  name      String
  weightG   Int
  cost      Decimal   @db.Decimal(10,4)
  portions  Portion[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  @@index([companyId])
}

model Portion {
  id            Int            @id @default(autoincrement())
  company       Company?       @relation(fields: [companyId], references: [id], onDelete: Cascade)
  companyId     Int?
  ingredient    Ingredient     @relation(fields: [ingredientId], references: [id])
  ingredientId  Int
  name          String
  weightG       Int
  cost          Decimal        @db.Decimal(10,4)
  snackPortions SnackPortion[]
  createdAt     DateTime       @default(now())
  updatedAt     DateTime       @updatedAt
  @@index([companyId])
  @@index([ingredientId])
}

model Snack {
  id            Int            @id @default(autoincrement())
  company       Company?       @relation(fields: [companyId], references: [id], onDelete: Cascade)
  companyId     Int?
  name          String
  imageUrl      String?
  snackPortions SnackPortion[]
  createdAt     DateTime       @default(now())
  updatedAt     DateTime       @updatedAt
  @@index([companyId])
}

model SnackPortion {
  id        Int      @id @default(autoincrement())
  snack     Snack    @relation(fields: [snackId], references: [id], onDelete: Cascade)
  snackId   Int
  portion   Portion  @relation(fields: [portionId], references: [id], onDelete: Cascade)
  portionId Int
  quantity  Int      @default(1)
  createdAt DateTime @default(now())
  @@unique([snackId, portionId])
  @@index([snackId])
  @@index([portionId])
}
```

## 5. LÓGICA DE NEGÓCIO IMPORTANTE

### Custo de Porção (calculado automaticamente)

```
porção.cost = (ingrediente.cost / ingrediente.weightG) * porção.weightG
```

### Totais de Snack (calculados em runtime)

```
snack.totalCost    = Σ(porção.cost × quantidade)
snack.totalWeightG = Σ(porção.weightG × quantidade)
snack.suggestedPrice = totalCost × 2 (margem de 100%)
```

### AddPortion no Snack

- Se porção já existe no snack → incrementa `quantity`
- Se não existe → cria com `quantity: 1`

### RemovePortion do Snack

- Se `quantity > 1` → decrementa
- Se `quantity === 1` → remove o registro

### CI/CD (GitHub Actions)

**Nota operacional (importante):** Ao sugerir comandos para iniciar containers com `docker compose`, use o NOME DO SERVIÇO (a chave sob `services:` em `docker-compose.yml`), por exemplo `docker compose up -d postgres`. O campo `container_name` configura apenas o nome do container Docker criado e NÃO é o identificador de serviço do Compose — usar `container_name` em comandos causa erro "no such service".

Pipeline com 3 jobs sequenciais:

1. **code-quality** → ESLint + TypeScript check (`continue-on-error: true`)
2. **test** → PostgreSQL service + Prisma migrate + Jest (depende do job 1)
3. **build** → `npx tsc` compilation (depende do job 2)

A CI usa seu **próprio PostgreSQL** (postgres:15, user:test) — completamente independente do Docker local.

## 9. REGRAS DE SEGURANÇA SaaS

**REGRA DE OURO: Empresa A NUNCA vê dados da empresa B.**

1. `companyId` SEMPRE vem do JWT decodificado (`req.user!.companyId`), NUNCA do body/params
2. Toda query ao banco DEVE filtrar por `companyId`
3. Ao criar recursos (ingrediente, porção, snack), incluir `companyId` no INSERT
4. Validação cruzada: ao criar porção, verificar que o ingrediente pertence à mesma empresa
5. Ao criar snack com porções, verificar que porções pertencem à mesma empresa

## 10. CONVENÇÕES DE DESENVOLVIMENTO

### Commits (conventional commits)

```
feat:      Nova funcionalidade
fix:       Correção de bug
docs:      Documentação
refactor:  Reestruturação sem mudar comportamento
test:      Testes
chore:     Config, deps, tooling
```

### Branches

```
main           → Produção estável
feature/*      → feat: funcionalidades novas
fix/*          → fix: correções
docs/*         → docs: documentação
chore/*        → chore: configurações
```

### Git Workflow

1. Criar branch: `git checkout -b feature/nome-descritivo`
2. Commits descritivos: `git commit -m "feat: add Company model to Prisma schema"`
3. Push: `git push origin feature/nome-descritivo`
4. Abrir PR no GitHub
5. CI/CD roda automaticamente
6. Merge para main quando validado

### Código TypeScript

- Usar tipos explícitos (nunca `any` sem justificativa)
- Interfaces em `src/types/`
- Services → lógica de negócio
- Controllers → validação + chamada ao service + resposta HTTP
- Helpers → funções utilitárias reutilizáveis

### Não commitar

- `.env` (usar docker-compose.yml para local)
- `node_modules/`
- `dist/`
- Arquivos de IDE (.vscode settings pessoais)
